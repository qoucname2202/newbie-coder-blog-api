# 02 — Business Rules

This document defines **domain rules** for the sample **TodoItem** aggregate. The database schema and entity exist; CRUD API and service layer are **planned but not yet implemented**. Rules marked **(planned)** apply when CRUD is built.

---

## Aggregate: TodoItem

### Field definitions

| Field | Type | Required | Rule |
|-------|------|----------|------|
| `Id` | `Guid` | Yes | Primary key; generated by application on create, not supplied by client on POST |
| `Title` | `string` | Yes | Non-empty after trim; maximum 200 characters (matches DB `nvarchar(200)`) |
| `IsCompleted` | `bool` | Yes | Defaults to `false` on create |
| `CreatedAt` | `DateTime` | Yes | Set once at creation; not client-writable on create/update |
| `UpdatedAt` | `DateTime?` | No | Set on every successful update; `null` if never updated |

### Source of truth

| Concern | Enforced by |
|---------|-------------|
| Title max length | EF configuration (`TodoItemConfiguration`) + service validation **(planned)** |
| Title not empty | Service validation **(planned)** — DB allows empty string unless app validates |
| Required columns | Database NOT NULL constraints |
| Index on `IsCompleted` | Database index for filtered list queries **(planned)** |

---

## Business rules catalog

### BR-001 — Valid title **(planned for CRUD)**

- Title must not be `null`, empty, or whitespace-only after trim.
- Title length must not exceed 200 characters.
- Violation → `BusinessException` with HTTP **400**.

```csharp
// Expected validation (service layer)
if (string.IsNullOrWhiteSpace(title))
    throw new BusinessException("Title is required.", 400);
if (title.Length > 200)
    throw new BusinessException("Title must not exceed 200 characters.", 400);
```

### BR-002 — Completion status **(planned)**

- `IsCompleted` accepts only `true` or `false`.
- Toggling completed ↔ pending is always allowed.
- No workflow states beyond completed/not completed (no "archived", "cancelled", etc.) unless explicitly added later.

### BR-003 — Delete behavior

- **Hard delete** — row is removed from `TodoItems` table.
- No soft-delete column exists in current schema.
- If soft delete is added later: new ADR required, migration required, update this document.

### BR-004 — Not found **(planned)**

- Any operation referencing a `Guid` that does not exist in `TodoItems` → `BusinessException` with HTTP **404**.

```csharp
var item = await repository.GetByIdAsync(id, ct);
if (item is null)
    throw new BusinessException($"Todo item '{id}' was not found.", 404);
```

### BR-005 — Timestamps

- **On create:** set `Id = Guid.NewGuid()`, `CreatedAt = DateTime.UtcNow` (or consistent app convention), `UpdatedAt = null`, `IsCompleted = false` unless explicitly set.
- **On update:** set `UpdatedAt = DateTime.UtcNow`; never modify `CreatedAt` or `Id`.
- Clients must not control `Id` or `CreatedAt` via request body on create.

### BR-006 — Paging for list **(mandatory for REST GET collections)**

Any GET returning multiple records **must** use pagination.

Using `PagingRequest` + `PagingDefaults`:

| Rule | Value |
|------|-------|
| Default page | `PagingDefaults.DefaultPage` (1) |
| Default page size | `PagingDefaults.DefaultPageSize` (10) |
| Minimum page | 1 |
| Maximum page size | `PagingDefaults.MaxPageSize` (100) |

Invalid paging → HTTP **400** + `ResponseCodes.ValidationError` (`00000201`).

### BR-007 — Sorting **(planned)**

`SortDirection` enum (`Asc`, `Desc`) exists in Core but is unused. When list API is added:

- Default sort: `CreatedAt Desc` (newest first) — recommended convention
- Document chosen default in `05-api-contracts.md` when implemented

## Authorization rules

### Current state

**No authentication.** All endpoints are publicly accessible. This is acceptable for local learning only.

### Future state **(planned — JWT)**

When authentication is added (see ADR-011):

- Each user should only read/write their own todos
- Requires `UserId` on `TodoItem` or separate ownership table
- New business rules and migrations will apply — update this document

## Exception → HTTP mapping

| Situation | Exception | HTTP status |
|-----------|-----------|-------------|
| Invalid input (title, paging) | `BusinessException` | 400 |
| Todo not found | `BusinessException` | 404 |
| Duplicate conflict (if added) | `BusinessException` | 409 |
| Unauthorized (future) | `BusinessException` or auth middleware | 401 / 403 |
| Unexpected error | Unhandled → middleware | 500 |

All JSON error responses use the standard envelope with `responseData: ""` and business code in `responseStatus.responseCode`. See [05-api-contracts.md](05-api-contracts.md).

---

## Validation layering **(planned)**

Recommended order when FluentValidation is added:

1. **FluentValidation** — request DTO shape and basic rules at API boundary
2. **Service layer** — business invariants (existence, ownership, timestamps)
3. **Database** — last line of defense (length, NOT NULL)

Do not rely on database alone for user-facing error messages.

---

## Agent notes

- Do not implement CRUD without following these rules
- Do not add fields to `TodoItem` without updating configuration, migration, and this document
- Distinguish clearly in code comments/docs between **implemented** and **planned** behavior
- When implementing CRUD, add tests for each business rule above
